MVP Design Proposal: Drink Roulette

Based on our decisions, here is the architecture and user flow for the first iteration.

1. The Core Flow (User Perspective)

1. Scan: Users scan a static QR code on the table (e.g., Table 4).
2. Join: They land on a web app, enter a nickname/emoji, and hit "Join Table".
3. Select: The first person to join ("Host") selects a pre-set round (e.g., "6 Tequila Shots - $40").
4. Ready: Host sees everyone who joined. Hits "Spin the Wheel".
5. The Spin: A 10-second CSS/Canvas wheel animation syncs across all connected phones.
6. The Reveal: The wheel stops on one person. Their phone turns red and prompts Apple Pay. Everyone else gets a "You Win!" confetti screen.
7. Fulfillment: Once paid, a ticket prints at the bar (or appears on a bartender tablet) for Table 4.

2. Architecture & Tech Stack

Frontend (The Web App)

  • Next.js or React SPA (Vercel)
  • Optimized for mobile Safari/Chrome (no app download)
  • CSS/Canvas for the 60fps wheel animation
  • Stripe Elements for Apple Pay / Google Pay

Backend (The Real-Time Engine)

  • Node.js + Socket.io server
  • Redis to manage table state (who is at Table 4? what is the current wheel state?)
  • Webhook endpoint to receive Stripe payment success and trigger the bar ticket.

3. The "State Machine" of a Table

IDLE → GATHERING (users joining) → SPINNING (wheel animating) → PAYMENT_PENDING (waiting for loser to pay) → PAID (ticket sent) → IDLE
