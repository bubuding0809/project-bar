MVP Design Proposal (Revised): Drink Roulette

Based on your feedback, we are integrating the game as an upsell/feature inside a standard QR ordering menu, rather than a standalone game-only app.

1. The Core Flow (User Perspective)

1. Scan: Users scan the QR code on the table (e.g., Table 4).
2. Menu View: They land on a standard digital menu (like TabSquare/Oddle) where they can browse individual drinks/food.
3. Game Discovery: A prominent floating button or top banner advertises "Play Drink Roulette 🎰".
4. Enter Game & Select: The first person to tap it ("Host") enters the game lobby. They select a pre-set round (e.g., "6 Tequila Shots - $40").
5. Friends Join: The Host's screen shows a "Waiting for friends..." state. Other people at the table see "Table 4 is playing Roulette - Join in!" at the top of their menu and tap to enter their name/emoji.
6. The Spin: Once enough people join, Host hits "Spin". The 10-second synced wheel animation plays.
7. The Reveal & Payment: The wheel stops on the loser. Their phone prompts Apple Pay. Others see a Win screen.
8. Fulfillment: Once paid, the pre-set order goes to the bar.

2. Architecture & Tech Stack

Frontend (The Web App)

  • Next.js/React SPA for both the menu and game views.
  • Standard e-commerce layout for the menu.
  • CSS/Canvas for the 60fps wheel animation.
  • Stripe Elements for checkout (standard cart + loser checkout).

Backend (API & Real-Time Engine)

  • Node.js + REST API for menu/catalog data.
  • Socket.io server for real-time table state (Is a game active? Who is joined?).
  • Redis to manage table state and sync users viewing the menu vs. in the lobby.

3. Table State Sync

Because the game sits alongside the menu, the backend must broadcast to all users at Table 4 when a game is initiated, so their UI can show the "Join Game" prompt dynamically while they are browsing drinks.
